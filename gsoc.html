Objective

Accessibility to any digital resource (websites, user interfaces etc.) depends on the way it is served to the user. Problems in accessing can be caused due to disability, illiteracy, digital illiteracy or aging. It is important that we consider user's ease of access. One way to address this problem is using preferences.

Preferences are choices of how the user wants the resource to be served . Some examples of digital preferences are: (1) Font/Text Settings - This includes font type, font size, text leading and so on (2) Theme Settings - Higher or low contrast (3) Accessibility Settings - Text to speech (speech rate, speech volume) or text for audio i.e. all sounds and audio information should be presented as text. (4)  Control Settings – User prefers mouse or keyboard for interaction. This also includes the specificity for on-screen or physical keyboard.

We start with collecting user’s digital preferences. Then we apply them across all the interfaces that user uses. This results in experiences that are consistent, familiar and consider user's ease of access.

To understand the idea take this example. You are reading this page with black text on white background. But, maybe you want less stress on your eyes and you prefer white text on black background. Now, that is your preference which makes accessing this page easy for you. This also illustrates a beautiful aspect of Inclusive design, that it provides benefits to everyone ,not just people with disabilities.

Thus, discovering user's preferences and applying them across digital resources is useful. This project aims to tackle the first part - developing an application that helps users to discover their preferences.

There have been attempts to address the problem of preference discovery. One implementations of this idea is First Discovery Tool (FDT) by Global Public Inclusive Infrastructure (GPII).

First Discovery Tool is like a wizard that takes you through different steps where you select your preferences. The ones shown here are language, text-to-speech, contrast, text-size.

Though First Discovery Tool fulfills the objective of collecting preferences. But, going through it feels like clinical evaluation. This does not align with the principles of inclusive design. This can be understood by the example of OXO potato peeler. The peeler allows someone with arthritis to hold a comfortable and steady grip while peeling, but it’s also ergonomic, allowing kitchen staff to avoid repetitive strain injury. This makes OXO peeler inclusive as both kitchen staff and someone with arthritis uses it in the same way. Similary, we don't want people to feel that preference discovery is just for people with disabilites. We want to encourage everyone to discover their preference. A wizard like FDT doesn't feel like it's designed for everyone.

❝ The goal is to get the user to think deeply about his/her prefs and make selections based on those insights and metacognition. First of all, in a social sense, we want everyone to be aware of and activated about their preferences -- building the culture of usability/a11y and autonomy. it's better to create tools that go for that directly, without burying the concept in a short-term pragmatic approach (like FDT). If the medicine is already sweet, we don't have to hide it in a cupcake, and what is sweeter than narcissism ("my needs and preferences are special and unique and it's fun to learn more about myself")? We need a way to add game-like features in First Discovery for motivating and engaging users, breaking away from a 'clinical evaluation' model, and generating buzz. ❞

— Jim Tobias, Inclusivity expert. Creator of FDT. From a discussion with him during the project.

Creating an application that does the following -

Make discovery of preferences fun We decided to create a game to help with preference discovery. As games provide a way of motivating and engaging users. The game was about a cat. The story goes as - Once lived a cat who loved her yarn ball. An evil rat stole the yarn ball. Cat chased the rat into the rathole. But, the evil rat was really quick and he escaped. The cat found some virtous rats inside the rathole they told her, "We saw the evil rat in the rat house. He has locked the yarn ball in the safe(locker). Each room of the rat house has a passcode in it. You would need to get all the passcodes to open the safe. When you go into the room you will find a tool, you can use the tool to find the passcode."

Phaser and Infusion Frameworks

Phaser is a game engine, and infusion is a JavaScript framework for creating applications that are highly accessible and reusable. Challenge for this project was to make a wrapper for Phaser using Infusion and using Phaser through infusion. To understand how these two work together in Discovery Cat, lets get a understanding of structure of Phaser -

Every game in Phaser has a main parent game object created using Phaser.game around which the game is built. Another important thing in a Phaser game are States. States are like scenes in a game like the loading screen, main menu, or any of the level. These states are the building blocks which are added to the game object eventually to make the game. Each states is a constructor function with prototype methods create, preload and update attached to them, which are used to define the logic and functionality of that scene. You can read about them more in the Phaser tutorial.

Infusion on the other hand is a framework made with intent of reusability in mind. The implementation unit of infusion is called a component. You can learn more Infusion here.

To make a wrapper for Phaser I created a infusion component which when instantiated, creates a Phaser.game object and stores it. Next I created each state for the game as a component and added these components as sub-components to the main component containing the Phaser game object. This way the sub-component can access the Phaser game object and also the game object can access these states and add them into itself to make the game work. This is the basic ideology behind the code of Discovery Cat, and might help you to understand its code more easily. You can find the code of the game below.

Exploring WebSpeech API

Web speech API enables browsers to handle voice data. It has two parts - one is voice recognition and other one is voice synthesis. For the project Discovery Cat we particularly required speech synthesis for adding the feature of text to speech(Screen Reader) to the game. The need to use it rose as the HTML5 game engines like Phaser uses canvas and normal screen reader cannot read the canvas. So I chose web speech API for text to speech support. If you want to know how I solved the issue you can find it in my blog on midterm evaluation.
