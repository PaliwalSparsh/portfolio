Objective

Accessibility to any digital resource(websites, user interfaces etc.) heavily depends on the way it is delivered to the user. It is important that it considers user’s ease of access.

Preferences are choices of how the user wants to consume the resource. Problems in accessing digital resources can be caused due to disability, illiteracy, digital illiteracy or aging. We can solve the problem by collecting the user’s digital preferences and applying them on all interfaces he would come across. This would result in consistent and familiar experience across all digital content that also consider his ease of access. Imagine presently you are reading this page with black text on white background but you would find white text on black background more comfortable(maybe less stress on eyes), now that is your preference which makes accessing this digital medium easy for you. So if we are able to discover an individuals preferences for accessing a medium(let it be digital here) and apply it across all the similar interfaces used by him, his experience would become seamless. This would be a use-case after discovering preferences but for this project we are interested in the former part of discovering preferences

Some of the digital preferences include -

1. Content Language - Language in which content is presented. (This also include sign language through symbols).

2. Font/Text Settings - This includes font type, font size, text leading etc.

3. Theme Settings - User prefers high or low contrast.

4. Accessibility Settings Text to speech (speech rate, speech volume), text for audio ie all sounds and audio information should be presented as text, normal audio volume/rate.

5. Control Settings – User prefers mouse or keyboard for interaction. This also includes the specificity for on-screen or physical keyboard.

Now that we understand knowing user’s preferences can be very beneficial. Next logical step, would be to collect these preferences from users. There have been amazing attempts to solve this problem, some of these solutions use Infusion Preference Framework. Preference framework enables us to make tools that can facilitate preference collection from users and apply them to all the interfaces. One implementations of this idea with Preference framework is First Discovery Tool of GPII.

Insert images of First Discovery Tool

Though First Discovery Tool fulfills the objective of collecting preferences. But, going through it feels like clinical evaluation. This is against what inclusivity is all about. The OXO potato peeler allows someone with arthritis to hold a comfortable and steady grip while peeling, but it’s also ergonomic, allowing kitchen staff to avoid repetitive strain injury. This makes OXO peeler inclusive as both kitchen staff and someone with arthritis uses it in a similar fashion.

Goal of the creator - the goal is to get the user to think deeply about his/her prefs and make selections based on those insights and metacognition. First of all, in a social sense, we want everyone to be aware of and activated about their preferences -- building the culture of usability/a11y and autonomy. it's better to create tools that go for that directly, without burying the concept in a short-term pragmatic approach. if the medicine is already sweet, we don't have to hide it in a cupcake, and what is sweeter than narcissism ("my needs and preferences are special and unique and it's fun to learn more about myself")? We need a way to game-like features in First Discovery as a way of motivating and engaging users, breaking away from a 'clinical evaluation' model, and generating buzz.

Requirements for the final product -

  1. Provides users understanding of preferences
  2. Makes discovery of preferences fun.
  3. Allows user to play with different preferences together.

PHASERPhaser will be used as HTML5 game framework for the project. Phaser is open source and has great community support, thus stabilizes itself to be an easy choice. Phaser has support for things like Phaser.touch library which will help us implement touch into the game.

PREFERENCE FRAMEWORK

We will be using the Infusion preference framework to make the Preference Editor, we will be making our custom prefEditor in preference framework, add required preferences to it accordingly, and manage CSS such that it aligns with the game design.PHASER AND PREFERENCE FRAMEWORKWe have discussed about the preference editor and phaser next point of discussion is how to use them together. We would be providing a wrapper/binding for Phaser so that we can use Phaser in accordance to Infusion practices. Also we will setup a model for the game to store all its states and preferences, so that we can access this model externally, this is another important infusion practice. We will be looking over all these implementation details in depth during the community bonding period.
