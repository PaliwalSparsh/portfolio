Exploration of Virtual Reality in Education

Duration: Jan 2020 - May 2020

Engagement Type: College Project

Team Size: 3

Checkout:

  1. Presentation Slides
  2. Code Repository

My role: Storyboarding, creating mockups, designing and animating 3D models. The programming work was divided among the team.

Objective

Traditional classrooms in India are usually centered around theoretical learning. The goal was to create a virtual classroom that provides both theoretical and hands-on experience to students.

Imagine a topic like volcanic eruptions, students are taught theoretically about such phenomenons. But, what if students can experience an immersive module on the topic. They can go below the earth's surface and get a sense of what it is to be magma, what are the factors that lead to the eruptions. This way students can explore, identify, and in cases experiment with the factors and learn about the topic.

We decided to create a module explaining students about Magnetic Resonance Imaging. What are the components of the MRI machines? How does MRI scanning work? The topic was ideal for the project as MRI machines are expensive and thus inaccessible for the purpose of learning. Also, experiencing MRI functioning which happens at the atomic level is not possible in traditional classrooms.

Deliverables

Developing a VR application to learn about MRI. The application will consist of 3 part -

  1. Students learn about the theory of MRI.
  2. Classroom converts into a space where students can experience an MRI machine - it's different components and how it is used in real-world.
  3. Finally, students get an overview of how things unfold at the atomic level. They can interact with different components during this part to learn about them.

Process

The project had two phases - designing and development.

Designing

The team started by exploring different VR alternatives. Eventually, we decided to use Google Cardboard as it was the most affordable solution in the market. Thus, ideal for students. Also, the only needed hardware to work with Google Cardboard was an Android Phone.

The workflow we followed had three steps. The first step was storyboarding the whole project. This helped us to communicate ideas and expectations within the team. The second step was creating mockup (sketches on paper), thinking over them and iterating until we got satisfied with the designs. The third step was developing the application.

Development

The tools we used to develop the project were -

  1. Cinema 4D to create 3D models and animations.
  2. Unity 3D to program the models and add behaviour.

Work in Unity 3D

Work done in unity is distributed over GUI and code. We’ll go over different sections we worked in the project to reach the final goal.

Firstly, we imported all the models in Unity and then arranged them to make the classroom.

Models imported to create the classroom scene

Google provides Unity Plugin that helps us to easily and efficiently set up everything required for creating VR experiences in Unity. Some useful components from the plugin include GvrAudio, GvrViewer, GvrRecticlePointer, GvrViewer, GvrAudioRoom.

Next, was adding behaviour to all the different models and making them interactive.

Adding motion to classroom fans.

Adding event trigger to the projector button.

Script flow or the program flow goes as - the first user uses the PLAY button (ProjectorSwitch.cs), video runs and gets over, and all the chairs and tables disappear from screen (afterProjectoin.cs), then chair goes back (chairMove.cs) and then all possible camera views (CameraView.cs) are shown from which user can see the models. These are the main scripts that run the program, apart from these other small scripts are used to add behaviour to elements like rotating the fan etc.

Conclusions

VR applications can provide a high level of immersion and Interactivity. They enable the analysis of micro and macro-level phenomenons. Also, they let users observe distant and even inaccessible places.

After finishing the project. We shared it with some of our friends and informally asked them about their experience. Most of them loved the application. One friend shared, "to see the machine go through the process helped him understand the topic clearly". Other expressed disapproval by saying, "the experience was not as great, the classrooms felt lifeless and even scary at times".

Few things that we can improve in the next iteration include - having multiple students attending the lecture at the same time, making systems more interactive so that students can control more variables in the system.

Reflections

We should have done evaluation on the project. We could have tested the final VR application with participants by dividing them into two groups. The first group would learn the topic traditionally. The second group would learn using the VR application. Then both the groups would give a test. The difference in performance on test would have indicated if the project was actually useful or not.
